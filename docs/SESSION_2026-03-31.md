# Session Notes — 2026-03-31

## Goal

Implement issue #9: create example template files for the `ProjectTemplate` system so users have concrete starting points when configuring their template folder in Settings.

---

## What Was Added

### Example Templates (`examples/templates/`)

Three JSON files covering distinct use cases:

- **`base-project.json`** — Generic default. Empty tags, 6 milestone-style tasks. Intended as the "start here" option.
- **`freelance-client.json`** — Client engagement lifecycle from proposal through sign-off. 8 tasks.
- **`personal-development.json`** — Self-directed learning arc with no business/sales terminology. 7 tasks.

An earlier iteration had `personal-dev-tracking`, `long-term-project`, and `poc-evaluation` — these were replaced mid-session after feedback that the tasks were too IBM/sales-specific and not useful as general examples.

### Bug Fixes

**`QuickCaptureView.swift` missing from Xcode project (`project.pbxproj`)**
The file existed on disk but was never added to the Xcode project, causing a build failure. Added `PBXBuildFile`, `PBXFileReference`, and group/sources entries manually.

**Security-scoped bookmarks for template folder access**
The app is sandboxed. The original implementation stored `url.path` (a plain string) in `UserDefaults` after the user selected a folder via `NSOpenPanel`. This lost the security scope token, so `ProjectTemplate.load(from:)` could never actually read the files. Fixed by:
- Storing `url.bookmarkData(options: .withSecurityScope)` alongside the path
- Adding `AppState.resolveTemplateFolderURL()` to resolve the bookmark at read time
- Calling `startAccessingSecurityScopedResource()` / `stopAccessingSecurityScopedResource()` in `ProjectTemplate.load(from:)`

**`@Observable` not tracking UserDefaults-backed properties**
`templateFolderPath` and `templateFolderBookmark` used computed get/set backed by `UserDefaults`. The `@Observable` macro only tracks synthesized stored properties, so SwiftUI views weren't re-rendering when the values changed (Clear button appeared to do nothing, template picker never appeared). Fixed by converting both to stored properties with `didSet` — matching the pattern already used by `themeMode`.

**Tag loading from template broken**
`NewProjectSheet` filtered template tags for `#` prefix before stripping it (e.g. `filter { $0.hasPrefix("#") }`). Template tags are stored as plain strings (`"personal"`, not `"#personal"`), so all tags were silently dropped on template apply. Fixed to `t.tags.joined(separator: ", ")`.

**`ExportService` warnings**
`iscOpportunityLink`, `gtmNavAccountLink`, and `oneDriveFolderLink` are non-optional `String` on `ExportedProject`, but the CSV encoder used `?? ""` on all three. Removed the redundant nil-coalescing operators.

### Settings UI

Added a template table to `SettingsView` showing loaded templates with columns: Name, Stage, POC (checkmark/dash), Tasks (count). The table appears when a folder is configured and templates are loaded. Window resized from `420×380` to `480×520` to accommodate.

---

## What Worked

- Security-scoped bookmark approach resolved the sandbox access issue cleanly
- Converting UserDefaults properties to stored + `didSet` is the right pattern for `@Observable` — matches how `themeMode` already works
- The template table in Settings gives useful immediate feedback when a folder is configured
- Subagent-driven implementation flow worked well for the pure-data tasks

---

## What Didn't Work / Needs Revisiting

**Template integration is shallower than intended.** The current system pre-populates stage, isPOC, tags, and creates initial tasks — but it doesn't touch checkpoints, notes, or any deeper project structure. The user indicated the feature needs a more thorough design before the next implementation pass.

**The template UX prompt question was left open.** The user wanted to explore making template selection more prominent during project creation (e.g. a dedicated first step rather than a picker buried in the form). This was not implemented.

**The POC template was removed** as part of the content redesign, on the rationale that it was too IBM-specific. This may need revisiting if the app's primary users are IBM SEs.
